# BLREC 3.0.0-beta.30

这是针对高光剪辑可靠性的公开测试版。本次修复针对的是可正常播放源录像在 FFmpeg 随机定位时出现的 FLV 索引和时间戳语义差异，不再把这类问题笼统归因于源文件损坏。当前候选已完成本地媒体矩阵和 NAS 真实录像门禁。

## 三级剪辑与产物验收

- 默认先执行低负载流复制，但不再把 FFmpeg 退出码为 0 当作成功；成片必须通过 H.264 编码、音频保留、500 毫秒时长容差、完整解码以及首尾视频包边界校验。
- 流复制未通过验收时自动转为 H.264/AAC；转码产物会连续核对首尾画面与源录像，而不是只看文件是否存在。
- 快速转码仍被 FLV 输入索引误导时，最后从文件头顺序解码转码。该路径较慢，但不依赖随机 seek，只在前两级失败时触发。
- 所有中间文件都会在失败后清理，最终文件只在全部验收通过后原子替换；审计日志记录实际策略和触发兜底的原因。

## 预检、重试与发布门禁

- 局部关键帧探测找不到目标点之前的关键帧时，会从文件头顺序扫描且只输出关键帧，覆盖 FLV 索引把探测位置跳到目标之后的情况，同时避免长录像积累全部视频包造成高内存占用。
- 音频先于视频开始且请求起点落在首视频关键帧之前时，只要所选范围内存在视频，实际起点会夹到首个可解码关键帧，不再拒绝整条任务。
- 预检缓存新增剪辑算法版本。升级后重试旧失败任务会重新探测源录像和关键帧，不再重复使用旧版本的预检边界。
- GitHub 发布流程新增不可跳过的真实 FFmpeg 回归，覆盖 MP4、碎片化 MP4、不同 FLV 索引、H.264/H.265、不同帧率、GOP、B 帧、音频状态和录制状态。
- NAS 发布门禁从 72 条符合条件的真实录像中均匀抽取 60 条，覆盖开头、25%、50%、75%、90% 五个位置和 5、10、15、20、30 秒五种片段时长。2026-07-23 最终验收为 300/300 完成、0 个失败、通过率 100%；其中 253 例通过流复制，47 例被边界校验拦下后自动转码成功，验证容器退出码为 0 且未发生 OOM，总耗时 3915.831 秒。

## 升级说明

本版本不新增数据库迁移、环境变量或目录挂载，可从 `3.0.0-beta.28` 或 `3.0.0-beta.29` 直接升级。升级前仍应备份 `/volume1/docker/blrec-next/config` 与当前 Compose 文件。
