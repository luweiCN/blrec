# 媒体库复用录制场次并在录制根同级永久保存

媒体库条目一对一关联现有录制场次，继续复用其分 P、预览、剪辑、投稿历史和本地删除能力；外部导入也创建内部场次与分 P，而来源由媒体库条目区分。永久文件保存在录制根同级的 `favorites/<storage-key>/`，容器部署将其独立持久化为 `/favorites`。

收藏先持久化文件移动计划。同一文件系统优先使用原子重命名；遇到跨挂载 `EXDEV` 时，在目标目录流式复制到受控临时文件，同步并原子落为目标文件后才删除源文件。数据库路径切换使用后续短事务，因此大文件复制不会长时间占用 SQLite 写锁。源、目标同时存在或目标已存在的中断状态可由同一计划恢复。

## Considered Options

- 独立的媒体库文件/任务模型会复制现有成熟的预览、剪辑和投稿状态机。
- 放在 `/rec/favorites` 可始终复用录制挂载，但不符合永久收藏与临时录像的目录边界要求。
- 扩展 `recording_sessions.source_kind` 需要重建被大量外键引用的核心表；媒体库条目的 `origin` 能以追加迁移表达真实来源。

## Consequences

- `source_kind='live'` 表示完整、多分 P 的源媒体形态；“系统录制或外部导入”由媒体库条目记录。
- 收藏只移动文件并增加媒体库成员关系，不复制场次或上传任务。
- 跨挂载收藏期间会短暂同时占用源和目标空间；复制完成前源文件始终保留。
- 媒体库成员不进入容量统计、低空间回收或任何投稿后保留策略，只能显式手动删除。
- 媒体库删除沿用场次本地删除语义，不删除 B 站稿件。
