# 高光剪辑可靠性门禁

## 上线标准

- 已建模的真实 FFmpeg 组合必须 100% 通过；不能带着已知失败发布。
- 发布前用 NAS 真实录像执行至少 300 次只读抽样剪辑，必须零失败、输出可完整解码。
- 线上滚动成功率目标不低于 99%；失败率按“完成或失败的剪辑任务”计算，取消和源录像已删除不计入算法失败。
- 任一输出只要首关键帧、结束边界、时长或音频不合格，就不能进入 `ready`。

300 次零失败只能把二项失败率的单侧 95% 上界近似压到 1%，且仍依赖样本具有代表性。因此它是发布门禁，不是“以后绝不会出现新问题”的证明。

## 2026-07 根因复盘

同一条 FLV 输入侧 seek 可能因索引布局落到前一个或后一个 GOP；输出侧 seek 在 stream copy 下也可能直接跳到下一个关键帧。FFmpeg 在这些情况下仍可能返回成功，所以只检查退出码和总时长不能证明片段正确。

问题连续复发的直接原因是：

1. 单元测试只断言 FFmpeg 参数，没有检查真实输出的首尾内容。
2. 唯一的真实 FFmpeg 测试依赖本地环境变量，CI 和发布流程默认跳过。
3. 旧时长容差为 `max(2 秒, 2%)`，会放过缺头、多尾和两者互相抵消的结果。
4. 重试重复执行同一条确定性命令，不能修复 seek 语义错误。
5. 前一次修复只验证了一个生产样本，没有覆盖相反的 FLV 索引行为和靠近文件开头的非零时间戳。
6. 已保存的预检结果过去没有算法版本；升级后重试可能继续复用旧算法留下的关键帧边界。
7. 音频流可能从 0 秒开始，而视频流的首关键帧在之后才出现；请求起点落在两者之间时，旧检查会错误地要求起点之前必须存在视频关键帧。

## 当前防线

剪辑先尝试低负载 stream copy，然后验证：

- 输出文件非空；
- 原输入有音频时输出仍有音频；
- 输出时长与计划范围一致；
- 输出首视频包就是预检选定的源关键帧；
- 输出最后视频包映射回源文件后位于计划结束边界内。
- 输出从头到尾可完整解码，不能只依赖 FFmpeg 退出码和容器时长。

局部关键帧探测如果没有找到目标点之前的关键帧，会从文件头顺序扫描且只输出关键帧，避免 FLV 索引直接把探测位置跳到目标之后，也避免长录像积累全部视频包造成高内存占用。

任何语义验证失败都会删除临时结果，并自动使用 H.264/AAC 精确转码兜底。快速转码结果还会逐帧核对连续的首尾画面与源录像边界，并再次执行完整解码；如果快速转码仍受输入索引影响，则最后从文件头顺序解码转码，不再依赖 seek 索引。超时或 FFmpeg 无法启动不会立即再执行一遍同类重任务。最终文件仍只在验收通过后原子替换；审计日志记录 `cut_strategy` 和 `fallback_reason`。

预检缓存带有剪辑算法版本。版本升级后，旧失败任务重试时会重新探测源录像和关键帧，不再沿用旧版本的预检边界。

如果请求起点早于视频流的首关键帧，但所选范围内确实存在视频，实际剪辑起点会夹到该首关键帧；只有整个所选范围都没有可解码视频时才拒绝任务。

GitHub 发布工作流中的 `Highlight media regression` 是非可选门禁。它自动生成并验证：

- MP4、碎片化 MP4、普通 FLV、直播式无元数据 FLV、带关键帧索引 FLV；
- H.264/H.265，25/30/60 fps，2/3/15 秒 GOP，0/2/3 个 B 帧；
- 有音频和无音频；
- 音频先开始、视频首关键帧延后；
- 文件开头、关键帧附近、中段、30 秒 seek 边界之后和文件尾部；
- `recording` 与 `finalized` 两条执行路径；
- 旧版兼容多分段拼接。

矩阵包含 160 个剪辑组合、10 个首尾画面对比、1 个音视频起始时间错位用例和 1 个真实多分段用例，共 172 例。每个输出还会完整解码，任何一个失败都会阻断发布。

## 明确边界

- 源录像已删除或关联丢失时无法恢复内容；重试会被拒绝并给出明确原因。
- 正在写入或刚结束且暂时无法探测的录像继续按瞬态错误退避重试。
- 真正不可读取的源文件不能靠剪辑器创造内容；这不属于 seek 算法成功率。
- 新建片段限定单录像分段。旧版多分段任务只在编码参数兼容时继续处理；不兼容的旧任务仍明确失败，不静默拼接错误媒体。

## 发布前命令

本地真实媒体矩阵：

```bash
BLREC_RUN_HIGHLIGHT_MEDIA_TESTS=1 \
  python -m pytest tests/bili_upload/test_highlight_cut_ffmpeg.py -q
```

NAS 抽样器必须把录像目录只读挂载，并把输出指向独立临时目录。300 例会从符合条件的录像中均匀选取最多 60 个源文件，并确保每个入选文件分别覆盖开头、25%、50%、75% 和 90% 五个位置，对应片段时长为 5、10、15、20 和 30 秒：

```bash
python scripts/validate_highlight_cuts.py \
  --source-root /rec \
  --output-root /validation \
  --cases 300
```

抽样完成后确认临时输出为空、一次性容器已删除、生产容器版本和四个既有挂载未改变。
