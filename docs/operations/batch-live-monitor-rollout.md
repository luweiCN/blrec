# 批量直播监控发布门禁

本文定义从批量直播监控 canary 到全部 58 个房间的发布门禁。任何一项未满足时都必须停留在当前阶段；不得通过增加账号、轮换代理、并发竞速或自动绕过风控来推进发布。

## 发布前检查

- 配置 `liveMonitor.mode` 为 `batch`，轮询间隔保持在已配置的 30–60 秒范围内。
- 批量状态请求必须使用匿名会话，不携带 Cookie、登录账号、鉴权信息或环境代理凭据。
- 确认运行状态接口可观测注册房间数、活跃 WSS、最近成功时间、快照年龄、缺项数、兜底请求数及熔断状态。
- 为每个阶段保存批量请求、单房间兜底请求、直播 start/end、WSS 连接和熔断日志。

## 阶段一：匿名单 UID canary

首次启用只能发送匿名、单 UID canary。不得把多个 UID 合并进第一次请求，也不得使用登录态重试。

通过条件：

- canary 返回可解析且与单房间匿名确认结果一致的状态；
- 遇到 HTTP 429 或 API `-352`、`-412` 时进入退避/熔断，没有并发重试或请求突发；
- canary 失败时不生成离线结论，不停止已有录制或 WSS；
- canary 成功后才允许进入 3–5 房间阶段。

## 阶段二：3–5 房间观察

选择 3–5 个具有代表性的房间，连续运行至少三天。每个入选房间都必须在观察期内完整经历至少一次开播到下播，未完成完整直播的房间不能计入通过样本。

三天观察期内必须同时满足以下全部条件：

- 所有离线房间的活跃 WSS 始终为 `0`，不是仅要求平均值为零；
- 从上游开播到本地发现的延迟始终处于已配置的 30–60 秒范围内；
- 每个唯一 `live_time` 对应的直播只产生一次 start 和一次 end，没有重复 start/end；
- 开播后只为实际直播的房间建立 WSS，下播确认后释放对应 WSS；
- HTTP 429 以及 API `-352`、`-412` 期间没有请求突发；除按退避节奏执行的单 UID canary 外，不得继续批量放量；
- 缺失条目、超时或异常响应不会制造假离线，也不会在熔断期间停止录制；
- 单房间兜底遵守 600 秒冷却窗口和同房间 singleflight，不因批量轮询重复触发。

任一条件不满足时，保持 3–5 房间规模或执行紧急回滚，不得继续放量。

## 阶段三：全部 58 房间

只有在 3–5 房间连续至少三天通过全部门禁，并且每个入选房间都完成一次完整直播后，才可启用全部 58 个房间。

放量后继续检查：

- 每轮最多两个批量请求，离线 WSS 保持 `0`；
- 发现延迟保持在配置的 30–60 秒内；
- start/end 仍按唯一 `live_time` 去重；
- 429、`-352`、`-412` 仍只触发退避、单 UID canary 和分阶段恢复，不出现请求突发；
- 熔断、缺项和兜底指标没有持续增长。

任何一项回退都必须停止扩大范围并评估是否执行紧急回滚。

## 紧急回滚到 legacy

`legacy` 只用于紧急回滚，不是常规运行或绕过批量接口风控的手段。切换 `liveMonitor.mode` 前必须先停止所有正在录制的任务；模式变更需要重启应用，只有重启完成后回滚才生效。

回滚到 `legacy` 会恢复旧实现的请求与 WSS 成本，包括为各房间维持的独立状态检查和连接开销。回滚后应确认旧监控链已恢复、批量协调器已停止，并记录触发回滚的指标和时间；不得把回滚后的资源成本误判为批量模式回归。
