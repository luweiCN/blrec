# 全链路请求性能整治设计

## 背景与目标

当前性能问题并非集中在单个列表。系统共有 103 个入站路由、91 处前端 HTTP 调用，以及房间状态、直播流、弹幕、投稿、通知等多类出站请求。新增的认证、上传、剪辑、网络管理和实时状态功能形成了共享资源竞争，导致原本很轻的页面请求也会排队。

本次整治覆盖浏览器与插件到 BLREC 的 JSON API、SSE、WebSocket、媒体请求，以及 BLREC 到 B 站、通知服务和更新服务的出站请求。目标是逐项审计、按证据修复，不为了追求指标改变业务语义，也不通过提高 B 站请求频率换取前台速度。

## 已确认的基线

NAS 当前有 47 个录制场次、43 个上传任务、82 个上传分 P 和 32,783 条上传分块。43 个上传任务均已审核通过且没有活跃后续分支，但实时采样仍每秒聚合全部分块：5 条 SQL 的中位耗时为 59.327 ms，最高为 161 ms。

录制或上传列表默认返回 20 条时执行 26 条 SQL；返回当前全部 43 条时执行 49 条 SQL。房间投稿策略列表同样存在逐策略查询账号的 N+1。所有管理员请求还会同步更新使用 DELETE journal 和 FULL synchronous 的认证数据库。

前端存在初始 GET 与 SSE `resync` 重复加载、筛选请求不能取消、全局上传任务集合变化触发分页列表重载，以及无关 SSE 事件进入 Angular Zone 的问题。录制与上传页面的冷路由依赖约为旧版的 3.8 倍 gzip 大小。

## 请求审计台账

仓库新增一份按路由组维护的审计台账。每条请求至少记录：

- 调用方、请求方法和规范化路由；
- 数据库读取、写入次数及耗时；
- 文件、子进程和外部网络依赖；
- 超时、重试、并发、缓存与幂等策略；
- 当前基线、目标预算、确认的问题、修复和验证结果。

破坏性接口不在 NAS 上批量压测。它们使用测试替身、底层管理器测试和单次受控验证。B 站出站请求主要通过已有日志、模拟响应和少量只读请求验证，禁止用高频探测制造额外风控风险。

## 共用请求基础设施

### 认证

会话认证仍在每次请求校验 token 和过期时间，但 `last_seen_at` 最多每 60 秒持久化一次；只有进入续期窗口时才更新过期时间。扩展 token 的 `last_used_at` 与审计记录同样限频。密码散列、初始化、改密和恢复操作移入专用线程并限制并发，避免 Argon2 阻塞事件循环。

认证数据库操作不得与业务数据库混用，且不能因为优化削弱注销、改密后撤销会话、CSRF 和暴力破解保护。

### 请求观测

为非流式 API 建立统一请求上下文，记录规范化路由、状态码、总耗时、响应大小，以及数据库、外部网络和子进程的调用次数与累计耗时。正常请求写 DEBUG；超过路由预算或失败时写结构化 INFO/WARNING。查询参数、Cookie、token、表单和媒体路径不得进入日志。

SSE、WebSocket 和媒体流分别记录握手/首字节、连接时长、事件或字节数、积压和断开原因，不以连接总时长判断“慢请求”。

### 实时通道

浏览器标签页仍只保留一个 SSE 连接。连接根据当前路由订阅 `tasks`、`network`、`upload_progress` 或 `highlight_progress` 主题，服务端只计算存在订阅者的主题。

上传实时查询只包含真正活跃、最近五分钟发生终态变化，或仍有活跃评论、弹幕、合集、修复分支的任务。上传进度使用专用轻量查询，不再复用完整任务详情。高光进度只返回活动或最近变化的片段，不能每秒扫描永久保存的全部片段。网络接口发现使用短期缓存，并在工作线程执行系统命令。

初始快照与首次 `resync` 合并；队列溢出和真实重连仍能触发完整同步。

## 入站接口优化

### 读取与列表

录制和上传列表改为轻量摘要。列表通过批量查询取得场次、分 P 汇总、上传摘要和弹幕计数，SQL 数不随页面条数增长；完整路径、未知弹幕、转码诊断和分 P 详情仅在打开抽屉时读取。列表阶段不执行文件 `stat`。

房间投稿策略用策略与账号的批量查询消除 N+1。合集名称使用短期缓存和单飞请求，房间列表不等待 B 站。磁盘容量、保留状态、高光时间轴和账号关联关系改为聚合查询或短期缓存，避免逐条文件与数据库读取。

为常用排序增加匹配索引。前导 `%LIKE%` 在数据量尚小时保留；只有基准证明搜索成为瓶颈后才引入 FTS。

### 前端请求与渲染

页面初始化只产生一次快照请求。筛选、分页和刷新使用可取消请求，旧响应不能覆盖新条件。复杂列表采用 OnPush 和独立行组件，只更新对象发生变化的行；模板中的重复筛选和线性查找改为预计算映射。

高光编辑器、播放器、片段库和大型设置弹窗按路由或打开动作加载。封面按可视区域加载并合并相同 URL 的并发请求。服务工作线程和路由预加载不再在首次进入时下载全部重型功能。

### 写操作与批量操作

纯数据库和配置动作使用单事务批处理；设置文件整批只保存一次。必须执行外部请求、停止录像、删除文件或等待后处理的动作使用 2–4 路有界并发，并保留逐项结果。

上传 Worker 不得为每条场次动作重复停止和启动。无需停止 Worker 的上传决定直接事务提交；确实要求暂停的整批操作只停止和启动一次。控制面操作不能无限等待审核、通知或其他远端工作完成；长操作返回可恢复的后台 operation 状态。

状态筛选统一为“待创建”“不自动上传”“取消上传”，对应计划上传但尚未建任务、跟随房间且自动投稿关闭、以及本场显式取消三种不同语义。

### 媒体与子进程

完成录像支持 Range、ETag 和条件请求；录制中的快照继续 `no-store`。文件打开、路径验证、FFprobe、FFmpeg 和 NAS 目录检查不得阻塞事件循环。探测与剪辑进入有界工作队列，避免多个请求同时耗尽 NAS CPU。

详情页所需的每分 P 高光数量使用轻量计数接口，不再加载完整时间轴。媒体接口分别验证首字节时间、拖动范围和持续吞吐。

## 出站请求优化

房间详情的一次刷新共享同一份 `getInfoByRoom`，不再分别为房间和主播重复请求。任务批量启动复用已取得的批量房间信息，并以小并发补充缺失详情。直播流探测与实际录像复用解析结果；旧流 URL 的验证请求必须关闭，优先取消额外探测 GET。

合集列表增加 30–60 秒 TTL 和请求单飞。投稿分类继续沿用现有 24 小时缓存。版本检查增加 15–60 分钟缓存及 stale-if-error。通知、Webhook、图片和更新检查复用连接、设置明确 connect/read/total timeout，并进入有界队列；Pushplus 使用 HTTPS。

上传、审核、评论、弹幕和通知从单一串行循环拆成职责清晰的 Worker，同时保留账号级写操作门禁。审核列表按账号共享，详情使用小并发和绝对截止时间。所有幂等重试遵守统一预算、指数退避和抖动，并尊重 `Retry-After`；非幂等请求继续使用现有“明确未发送/结果未知”保护。

弹幕、状态轮询、直播流和 UPOS 维持既定线路策略。性能优化不能把固定上传拆到多条线路，也不能提高房间轮询或弹幕发送频率。

## 分批实施

1. **共用止血**：请求观测、认证限频、订阅驱动 SSE、活跃任务查询、网络接口缓存。
2. **热读取路径**：房间、录制、上传、片段、容量和策略接口的 N+1、摘要/详情和索引优化。
3. **写操作与媒体**：批量事务、Worker 生命周期、后台 operation、媒体缓存与有界子进程。
4. **出站链路**：重复请求、连接池、缓存、超时、队列拆分和统一重试预算。

每一批必须先写能复现原问题的测试，再实现修复，并在合并后运行相关 Python、Angular、类型、lint 和生产构建。每批可独立部署 NAS，使用请求日志和数据库规模复测，上一批指标不允许在下一批回退。

## 性能预算与完成条件

- 当前 NAS 暖状态下，普通内存 GET p95 小于 25 ms，常规本地数据库 GET p95 小于 100 ms。
- 房间、录制和上传列表在 20 条分页下 p95 小于 150 ms，进入页面只发一次列表请求。
- 录制与上传列表 SQL 数保持常数级，列表文件 `stat` 数为 0。
- 没有相关 SSE 订阅者时，上传、高光数据库查询和网络接口子进程为 0。
- 配置型批量操作不随条目数量线性重启 Worker；58 个纯本地配置变更目标小于 2 秒。
- 媒体请求首字节 p95 小于 150 ms，完成文件支持缓存与范围请求。
- 所有外部请求都有明确总超时；通知和 Webhook 故障不能阻塞上传或无限积累任务。
- 请求审计台账覆盖全部 103 个入站路由和主要出站操作，所有确认问题都有测试、修复或明确的“无需修改”证据。

## 非目标

- 不重写 FastAPI、Angular、SQLite 或现有录制核心架构。
- 不为了跑分删除安全校验、持久化恢复和结果未知保护。
- 不对 B 站进行压力测试，也不扩大账号请求预算。
- 不在没有基准证据时引入 Redis、消息队列或新的外部服务。
