# 分 P 增量预上传与时间轴修正设计

## 目标

已启用投稿的直播间在某个分 P 完成后立即预上传该文件，不再等待整场直播结束。预上传任务直接出现在上传任务列表中，并与正式上传、投稿和审核状态明确区分。直播结束后只上传尚未完成的最后分 P，再使用结束时的最新投稿配置提交一个包含全部分 P 的稿件。

本轮同时修正高光剪辑时间轴的轨道对齐、操作浮层裁剪和悬浮虚线间歇消失问题。场次识别规则保持不变。

## 已确认的现状

当前 Python 实现只查询 `live_end_time IS NOT NULL` 的场次。它在下播后创建 `upload_jobs`，一次性写入全部 `upload_parts`，依次上传后立即提交稿件。因此较早完成的 P1 会一直等待最后一个分 P 和整场下播。

Java 参考实现由 `RecordEventFileClosedService` 在分 P 文件关闭后调用异步上传；最终投稿流程跳过已经上传成功的分 P，只等待或补传剩余文件，再提交一个稿件。本设计复用该业务时机，但沿用本项目已有的数据库租约、分块进度和断点续传机制。

## 预上传任务生命周期

### 创建与展示

录制中的分 P 完成后处理、文件身份可读取且通过稳定窗口后，协调器读取当前场次决定和房间投稿规则。需要投稿且账号可用时创建一个可见的预上传任务：

- 数据库继续使用现有 `waiting_artifacts`、`ready` 和 `uploading` 状态；
- 新增 `preupload_finalized` 标记，区分仍在录制的预上传任务和已经冻结配置的正式任务；
- 迁移时既有上传任务的 `preupload_finalized` 默认为真，不改变历史任务行为；
- API 提供面向用户的派生状态，不直接暴露内部状态组合；
- 上传任务列表立即出现该任务。

用户可见状态为：

- `录制中 · 正在预上传`：存在待传或正在传输的已封口分 P；
- `录制中 · 已预上传，等待新分 P`：当前已封口分 P 均上传完成，直播仍在继续；
- `录制中 · 预上传已暂停`：账号、网络或文件错误已触发现有暂停机制；
- 下播后沿用 `正在上传`、`正在投稿`、`等待审核` 等正式状态。

预上传进度显示已完成的分 P 数量和当前文件传输进度。直播未结束时不显示虚假的总分 P 数量。

### 新分 P 与投稿

协调器周期性同步同一场次中新增的 `recording_parts`。只追加 `artifact_state='ready'` 且稳定的分 P；已经存在的 `upload_parts` 不重复创建。任务进入 `ready` 后复用 `UposUploader` 上传新增分 P，成功后回到等待新分 P 状态，不调用稿件提交接口。

直播结束时重新读取场次覆盖设置和房间投稿规则，并执行一次原子化最终确认：

1. 生成包含全部有效分 P 的最终投稿快照和分 P 标题；
2. 将最后尚未同步的分 P 加入任务；
3. 将 `preupload_finalized` 设为真；
4. 上传器跳过已有 `remote_filename` 的分 P，仅上传剩余部分；
5. 所有分 P完成后只提交一次稿件。

只有最终确认完成的任务才能进入 `submitting`。即使“直播结束”和“P1 上传完成”并发发生，也不得使用临时快照提前投稿。

## 配置变更规则

直播结束时的配置是最终权威来源：

- 修改标题、简介、分区、标签、可见性、合集、封面或定时发布，不需要重新上传视频文件；最终投稿使用最新值。
- 关闭自动投稿、将本场设为不上传或删除任务时，停止预上传并删除本地任务状态，不向 B 站提交稿件；已传但未投稿的远端临时文件交由 B 站过期清理。
- 重新开启投稿时，可针对现有已封口分 P重新创建预上传任务。
- 投稿账号不变但凭据刷新时，保留已确认分 P，未完成部分使用新凭据继续。
- 投稿账号发生变化时，旧账号的远端文件不安全复用；清空预上传文件引用和分块状态，使用最终账号重新上传全部分 P。

录制期间允许编辑本场投稿设置。只有 `preupload_finalized=true` 后，投稿快照才进入现有不可变约束。

## 重启、失败与删除

分块确认、远端文件名和租约继续写入 SQLite。容器重启后：

- 已确认分块和分 P 不会重复上传；
- 正在传输的分块按现有 UPOS 恢复规则继续；
- 仍在直播的任务恢复为预上传状态；
- 已下播但尚未最终确认的任务先完成最终确认，再继续上传和投稿。

单个分 P 上传失败沿用自动重试和暂停规则。用户主动删除预上传任务时，停止当前租约并删除任务、分块记录及其本地文件（按现有删除确认语义），不删除 B 站稿件，因为此时尚未产生稿件。仅关闭自动投稿只取消预上传状态，不删除录像文件。

关键阶段写入审计日志：任务创建、分 P加入、预上传开始/完成、等待新分 P、最终配置冻结、账号变化重传、预上传取消、重启恢复和正式投稿。

## 时间轴布局与浮层

时间轴使用统一的轨道几何变量，避免不同元素各自维护 `top`、`bottom` 和 `height`：

- 主轨道与蓝色待编辑范围共用同一顶部和高度，蓝框完整套在主轨道上；
- 新增明确的已创建片段轨道，绿色范围完整落在第三轨；
- 播放头、边界手柄和悬浮参考线跨越所有轨道；
- 高光点仍直接位于时间轴上。

片段操作面板改用 Angular CDK connected overlay，锚定当前点、边界或范围中心并渲染到全局 overlay 容器。它必须自动避让视口边缘，且不受工作台 `overflow: hidden` 裁剪；全屏模式也必须可见。

高光点目前使用 `.marker-pin::after` 渲染悬浮说明，同样会被黑色工作台边界裁剪。该说明改用项目已有的 `nz-tooltip` 全局浮层，并保留高光名称和时间；靠近页面左右边缘时由 overlay 自动翻转或平移。删除原伪元素说明，避免出现两份提示。

当前拖动依赖 Pointer Events，悬浮虚线却依赖兼容的 `mousemove`；点击时的 `preventDefault()` 和 pointer capture 可能使部分移动过程不再产生对应的鼠标事件。悬浮与拖动统一改用 Pointer Events：指针位于正常时间轴区域时持续更新虚线和时间，只有真正离开时间轴才清除。虚线层级高于蓝色和绿色范围，低于可点击手柄与操作浮层，避免已经渲染却被片段范围遮住。不对录制中的不稳定尾部新增悬浮展示要求，仍保持该区域不可创建裁剪边界。

## 场次判定

继续以 B 站 `room_id + live_start_time` 生成场次键。相同标题不能用于合并场次。已调查的“诠释者BP”记录中：

- 第一场为 19:40:12 至 19:51:59；
- 第二场为 20:07:03 至 21:43:39；
- 两场相隔约 15 分钟，具有不同场次键、上传任务和 BVID。

因此这两条数据不是重复任务，不做合并或删除。

## 验证

后端自动化测试至少覆盖：

- 开放场次的 P1 稳定后创建可见预上传任务并完成上传，但不投稿；
- P2 加入后只上传 P2；
- 下播后使用最新配置，仅提交一次包含 P1、P2 的稿件；
- 关闭投稿取消预上传且不提交；
- 非账号设置变更复用已上传分 P；
- 账号变更清空远端引用并重传；
- 上传中重启后恢复分块进度；
- 并发下播不会使用未最终确认的快照投稿。

前端测试至少覆盖：

- 两种预上传状态、实时进度和正式状态切换；
- 蓝色范围与主轨道计算样式一致；
- 绿色范围位于独立已创建轨道；
- 操作面板位于 CDK overlay 容器且不被工作台裁剪；
- 高光点说明位于全局 tooltip 容器，靠近左右边缘时仍完整可见；
- 指针经过空白、蓝色范围和绿色范围时均可靠显示悬浮虚线；拖动结束后继续移动也不会丢失，只有离开时间轴才隐藏。

线上验收选择一个会产生至少两个分 P 的直播间，核对日志中 P1 的 `upload_part_completed` 早于 `recording_finished`，并确认下播后只上传最后一个分 P 且只产生一个 BVID。
