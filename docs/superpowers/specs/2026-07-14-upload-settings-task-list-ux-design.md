# 投稿设置与上传任务界面设计

## 目标

投稿设置应直接说明操作结果，不使用“规则与账号”“策略”等抽象概念，也不靠大段说明弥补命名。上传任务应是可分页浏览的紧凑列表，详细文件和弹幕信息不在列表中全部展开。

## 投稿设置弹窗

弹窗只保留三组明确标题：“稿件内容”“发布选项”“审核后处理”。顶部直接放置以下字段，不再增加分组标题：

- “自动投稿”：开启后，新直播结束时创建上传任务；关闭后不再创建新任务，已有任务不受影响。
- “投稿账号”：选项改为“使用主账号”或“指定账号”，不再称为账号策略。

“稿件内容”包含标题、简介、分 P 标题、投稿分区、标签、稿件类型和转载来源。“发布选项”包含可见范围、发布动态、动态内容、禁止转载、允许评论、仅展示精选评论和允许弹幕。“审核后处理”包含发布 SC/上舰索引评论和回灌直播弹幕。

负向的“关闭评论”“关闭弹幕”改成“允许评论”“允许弹幕”，提交时再转换为后端现有反向字段。“精选评论”改为“仅展示投稿账号精选的评论”，并作为评论的从属选项。关闭评论时同时关闭精选评论和索引评论；关闭弹幕时同时关闭弹幕回灌，避免产生无法保存的组合。

删除泛泛的段落，只保留字段校验、依赖关系和不可逆风险所需的信息。保存按钮除请求进行中外保持可用；点击后在具体字段附近显示“请选择投稿分区”等错误，不再静默禁用。

## 投稿分区

B 站当前 Web 投稿的 `tid` 使用子分区 ID。一级分区仅负责展开，二级叶子节点才可选择。前端把一级节点标记为非叶子、二级节点标记为叶子；选择后输入框显示“一级分区 / 二级分区”并立即更新草稿。

保存时后端使用当前投稿账号对应的分区目录再次确认 `tid` 属于可见子分区。无效或已下线的分区返回明确错误，不提交一级分区 ID，也不通过创建测试稿件探测接口行为。

## 上传任务列表

页面只保留一个“上传任务”标题。刷新按钮放在页面标题右侧，删除“上传任务列表”和“按同一场直播归集”等重复说明。

主体改为表格，每场直播一行，显示：直播标题与房间号、开始时间、录制概要（分 P 数/时长/大小）、录制状态、投稿状态、投稿账号和“查看详情”。底部分页默认每页 20 条，可切换 20/50/100 条。

后端列表接口增加 `offset` 和 `total`，保留现有 `limit`，每次只读取当前页。点击“查看详情”打开右侧抽屉，复用当前场次数据展示封面、文件、各分 P 上传状态、稿件编号、弹幕回灌进度和需要人工处理的异常。常规说明不重复显示；只有结果未知、可能重复发送等风险操作保留警告。

## 验证

后端测试覆盖子分区校验、总数和偏移分页。前端测试覆盖分区叶子选择与回填、保存错误提示、正向互动开关映射、依赖选项联动、表格列、分页请求和详情抽屉。最后运行完整 Python 测试、Angular 无头测试、相关 lint、生产构建，并在本机页面实际选择分区、保存设置和翻页。
