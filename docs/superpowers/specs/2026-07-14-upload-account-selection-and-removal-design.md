# 主账号、投稿账号与账号移除设计

## 目标与边界

本设计补充 `2026-07-12-builtin-bilibili-upload-design.md` 中的账号映射与管理规则，明确三类相互独立的行为：读取请求可以在确认凭据失效后使用备用账号；投稿、评论和弹幕回灌必须固定身份；账号移除必须先展示并处理关联关系。

本次同时修正主账号切换的运行时行为。切换 Cookie 只更新后续 API 请求和未来 WebSocket 重连所用的请求头，不主动断开仍在正常接收弹幕的连接。

## 账号语义

- **主账号**：为新建房间投稿配置提供默认身份，也供需要登录的房间详情和画质查询优先使用。
- **房间投稿账号策略**：可选择“跟随主账号”或“固定账号”。新配置默认跟随主账号。
- **上传任务账号**：创建上传任务时解析为确定的 `account_id`，并作为该稿件上传、投稿、审核查询、自动评论和弹幕回灌的共同身份。

主账号是默认策略，不是上传任务执行时动态读取的指针。切换主账号不会修改已经创建的上传任务。

读取链路与写入链路分离：房间读取请求在主账号被再次校验为失效后，可以临时使用其他可用账号，但不会改变主账号标记，也不会影响任何上传任务绑定。所有写操作禁止自动回退到其他账号。

## 房间策略与任务快照

`room_upload_policies` 增加账号模式：

- `primary`：不保存固定账号；创建任务时解析当前有效主账号。
- `fixed`：保存一个处于可用状态的 `account_id`。

现有房间策略迁移为 `fixed`，避免升级后静默改变投稿身份。新建策略默认为 `primary`。

创建 `upload_jobs` 时必须把解析后的账号写入现有非空 `account_id` 字段。任务只允许在尚无任何远端副作用时手动改绑账号，即任务仍处于等待制品、就绪或安全暂停状态，且所有分 P 均未创建预上传会话。出现预上传、分块上传、投稿提交、结果未知、审核、评论或回灌记录后，账号锁定。

同一 UID 的凭据刷新或重新扫码只更新该账号的凭据版本，不算切换账号，任务可以继续执行。账号失效时任务暂停并显示原因；用户应重新登录同一 UID。若必须换号，只能在满足无远端副作用条件时显式改绑，否则需取消并重新创建任务或进入人工处理。

## 主账号切换

点击“设为主账号”先打开确认弹窗并加载影响预览：

- 跟随主账号的房间数量及可展开的房间列表；
- 固定绑定账号的房间数量，并明确其不受影响；
- 已创建上传任务数量，并明确其不会改绑；
- 房间读取请求将开始使用新账号；
- 正常工作的弹幕 WebSocket 不会被重启。

确认后，后端在本地事务中更新主账号选择，再更新录制任务内存中的 Cookie 请求头。接口不等待弹幕 WebSocket 断开或重连，因此不应出现随活跃房间数增长的 loading 时间。现有连接继续收取弹幕，下一次自然重连时使用最新有效 Cookie。

## 账号移除而非物理删除

管理页提供“移除账号”，不直接删除数据库行。账号可能被历史稿件、评论、弹幕和审计记录引用，因此最终状态为 `archived`：保留账号 ID、UID、昵称和历史关系，清除可用于登录的凭据，使其不能再发起请求。同一 UID 以后重新扫码时复用并恢复原账号记录。

点击移除后先展示关联关系：

- 是否为当前主账号；
- 固定绑定该账号的房间及数量；
- 尚无远端副作用、允许迁移的上传任务；
- 正在上传、投稿、结果未知或执行后续写操作的阻塞任务；
- 已完成、拒绝或已取消的历史任务数量。

历史任务只用于说明，不迁移也不阻塞；它们继续引用归档账号。存在阻塞任务时禁止移除，并提供任务链接供用户先完成、取消或人工裁决。

## 移除时的关联处理

无阻塞任务时，弹窗提供三种明确选择：

1. **改为跟随新主账号（默认）**：关联房间改为 `primary` 模式；尚未开始远端操作的任务立即改绑为删除操作完成后的新主账号。
2. **固定切换到指定账号**：用户选择一个可用账号，关联房间改为 `fixed` 模式，允许迁移的任务同时改绑该账号。
3. **不迁移**：关闭关联房间的自动上传，允许迁移的任务进入暂停状态并保留人工处理入口。

“跟随新主账号”不能表述为“不切换”，因为它会改变未来任务及部分待处理任务的身份。

若移除的是当前主账号且仍有其他可用账号，用户必须先在同一弹窗选择新主账号。若没有其他账号，只允许“不迁移”：删除主账号选择、关闭关联房间自动上传并暂停待处理任务。匿名批量状态查询和核心录制继续工作。

服务端在执行前重新查询关联关系，并用一个数据库事务完成主账号替换、房间策略迁移、允许迁移的任务更新、账号归档、凭据清除和管理审计。若预览后新出现阻塞任务，返回冲突并要求前端刷新影响预览，不能部分执行。

## 管理接口与页面

后端提供只读影响查询，返回脱敏摘要、房间 ID/名称、任务 ID/状态和是否允许迁移，不返回 Cookie、token 或凭据版本内容。主账号切换与账号移除分别使用该摘要渲染弹窗。

账号移除请求包含关联处理模式、可选的固定替代账号和可选的新主账号。后端不信任前端传入的统计结果，只接受选择并自行重新验证。非法组合、不可用替代账号和新增阻塞任务返回 `409`。

页面默认只展示数量；房间或任务较多时按分类展开，避免 58 个房间一次铺满弹窗。危险操作按钮使用“移除账号”，确认文案明确说明“登录凭据会被清除，历史投稿记录仍会保留”。归档账号默认不出现在可选账号列表中。

## 错误与一致性

- 主账号或固定账号不可用时，不创建新的上传任务，并在房间策略处显示阻塞原因。
- 写操作遇到身份失效时暂停原任务，不选择备用账号。
- 已有远端副作用的任务不能通过账号移除流程改绑。
- 结果未知的投稿、评论或弹幕任务必须先人工裁决。
- 评论和弹幕回灌始终继承 `upload_jobs.account_id`，不得单独选择账号。
- 账号归档不会停止录制，也不会断开仍可工作的弹幕 WebSocket。

## 验证

后端测试覆盖：房间策略两种模式、任务创建时账号快照、切换主账号不修改旧任务、写操作不回退、可迁移与阻塞任务分类、三种移除方式、当前主账号替换、无备用账号处理、事务回滚、并发新增阻塞任务、凭据清除及同 UID 重新激活。

运行时测试断言主账号切换只更新 Cookie 请求头，不调用弹幕客户端重启；自然重连仍使用新请求头。前端测试覆盖两个影响弹窗、分类展开、替代账号选择、无备用账号限制、阻塞提示和服务端冲突后的重新加载。最后运行完整 Python/Angular 测试、静态检查和生产构建。
