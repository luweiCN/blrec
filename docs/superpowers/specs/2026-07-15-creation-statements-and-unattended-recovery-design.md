# B 站创作声明与无人值守恢复设计

## 目标

投稿设置与 B 站当前创作中心保持一致；录制进程因重启、崩溃或网络异常中断后，由程序自动判断可用文件并继续后续流程，不产生需要用户逐项确认的状态。

## 投稿模板与创作声明

新建投稿设置采用 `biliupforjava` 的默认内容：标题为“【直播回放】【主播】直播标题 时间”，简介和动态包含“直播录像”及直播间链接，标签为“直播回放,主播,分区”，分 P 标题为“P序号-分区-时间”，默认分区为 21。所有字段使用现有 Liquid 模板语法，标签也参与渲染。已有设置保持用户保存过的内容。

投稿账号对应的 `/x/vupre/web/archive/pre` 响应同时提供分区与 `neutral_mark.mark_list`。前端直接展示该列表，不写死声明文案或 ID。当前列表包含“内容无需标注”“含 AI 生成内容”“含虚构演绎内容”“内容含营销信息”“个人观点，仅供参考”和“内容为转载”。另提供“内容为自制：未经作者允许，禁止转载”授权声明；转载与该授权互斥，选择转载时必须填写来源。

策略保存 `creation_statement_id` 和 `original_authorization`。提交时按 B 站当前规则派生：转载使用 `copyright=2`；非转载且勾选授权使用 `copyright=1`；其余使用 `copyright=3`。仅授权声明开启时发送 `no_reprint=1`，并发送 `creation_statement: {id}`。投稿接口切换到当前 Web 端 `/x/vu/web/add/v3`，使用任务绑定账号的 Web Cookie 与 CSRF，不回退或轮换账号。旧策略迁移时，自制映射为“内容无需标注”，转载映射为“内容为转载”，保留原有禁止转载含义。

## 无人值守录制恢复

新增轻量文件探测：使用 `ffprobe` 读取一个视频包并确认存在视频流，同时获取文件大小和可用时长。探测不扫描整段录制，避免大量房间重启时长时间阻塞。

启动恢复按以下顺序处理录制中、后处理中、旧“需要确认”和成品丢失的分段：

1. 优先使用可解析的成品文件。
2. 成品不存在或不可解析时，回退到可解析的原始录制文件。
3. 两者都不可用时自动标记为失败或缺失，并记录原因；不要求人工确认。

后处理失败同样回退原始录制文件。同一场直播只要至少一个分段可用，就关闭归集并仅为可用分段创建上传任务；坏分段保留诊断记录但不阻塞其他分段。没有任何可用分段的场次自动标为已跳过。

因应用停止而取消的场次先保留短暂合并窗口，直播迅速恢复时继续原场次和分 P 编号；超过窗口仍未恢复则自动归集。若归集并创建上传任务后同一直播再次恢复，创建续录场次，确保新文件不会写进已经冻结的上传任务。

历史 `manual_review` 数据在下次启动时按同一规则自动消化。前端不再显示“需要确认”；失败或缺失只作为运行结果和日志展示。

## 界面收尾

投稿设置用一个“创作声明”控件替代旧“稿件类型”和独立禁止转载开关。上传任务刷新不再重新打开已关闭抽屉；文件列表只显示文件名，悬浮显示完整路径；分页使用独立、紧凑的 NG-ZORRO 分页栏。

## 验证

后端测试覆盖旧策略迁移、动态声明目录、Web 投稿签名与负载、标签模板、可用文件恢复、坏文件排除、后处理回退、取消场次自动归集和续录。前端测试覆盖默认模板、声明互斥与来源校验、刷新不打开抽屉、文件名缩略和分页交互。最终使用现有 11.18GB 中断 FLV 做只读探测，并重启本机服务确认历史“需要确认”自动转为可上传。
