# BLREC 内置 B 站投稿、弹幕回灌与自动评论设计

## 1. 背景与目标

BLREC 当前负责房间状态监控、直播录制和弹幕 XML 落盘，投稿依赖外部工具或 Webhook。首期改造将在同一仓库和 Docker 镜像内增加纯 Python 投稿链路，使录制完成后的文件能够直接进入可恢复的投稿任务，并在稿件审核通过后执行弹幕回灌和 SC/上舰索引评论。

目标包括：

- 使用扫码登录获得可正常刷新的账号凭据；
- 将一场直播的多个录制文件组织为一个多 P 稿件；
- 在管理页面配置房间投稿规则并查看全过程状态；
- 将 BLREC XML 中通过过滤的弹幕全部写入对应视频分 P；
- 自动发布并置顶 SC/上舰时间索引评论；
- 降低无效查询和并发写入，遇到风控时可靠暂停并续跑；
- 保证新增功能失败时不影响 BLREC 的核心录制能力。

## 2. 已确认范围

首期包含账号管理、投稿、多 P、审核同步、弹幕回灌、自动评论、管理 API/UI、自动化测试和线上灰度验证。自动评论使用确定性模板，不接入 AI。首期不包含云剪辑、高能片段、微信推送、自动删除或移动录像、账号池轮换、代理切换及其他规避风控的机制。

实现以 `mwxmmy/biliupforjava@a366e4f1f86bfd1c69a9b6cc66c372d2f6da7e1e` 的业务行为和 `biliup/biliup@18c5bf086e943e07e9d88a905d2e5d407d6305bb` 的登录、刷新及投稿协议为固定研究基线，但按 BLREC 的异步 Python 架构重新实现，不引入 Java 或 Rust 运行时。前者采用 Apache-2.0，后者采用 MIT；移植代码时保留必要的许可证和来源声明。B 站 API 备份仓库只作为协议研究资料，不能视为稳定或官方接口保证。

## 3. 现状与约束

BLREC 使用 Python 3.8+、FastAPI、`aiohttp`、RxPY 事件中心和 Angular 前端。录制过程已经发布 `RecordingStartedEvent`、`VideoFileCompletedEvent`、`DanmakuFileCompletedEvent` 与 `RecordingFinishedEvent`。当前项目没有业务关系数据库，设置主要写入配置文件；因此首期新增一个内置 SQLite 数据库，专门保存账号、投稿和后处理任务。现有录制设置格式与行为保持兼容。

部署继续使用一个仓库、一个 Docker 镜像和默认单容器，不要求 Redis、Celery 或外部数据库。后台工作器和 Web API 运行在同一应用生命周期内，但通过持久化队列隔离，网络长任务不得阻塞录制协程或 HTTP 请求。

### 方案选择

评估过三种实现方式：继续通过 Webhook 调用外部上传器，改为 BLREC 加 Java 边车，以及在 BLREC 内纯 Python 实现。Webhook 的部署和状态分散，无法自然共享账号、任务和管理页面；Java 边车可以复用较多代码，但需要维护第二套运行时、数据库模型和故障恢复。最终选择纯 Python 内置方案，以较高的首期移植成本换取单一部署、统一状态和长期可维护性。

## 4. 总体架构

新增组件及职责如下：

- `AccountManager`：扫码登录、凭据刷新、Cookie 与稳定设备标识维护、账号健康检查；
- `UploadCoordinator`：订阅录制事件，建立录制会话并将完成文件组织为投稿任务；
- `UploadWorker`：预上传、分块上传、断点恢复、多 P 提交和结果持久化；
- `ReviewWatcher`：按账号批量、低频同步待审核稿件；
- `DanmakuImporter`：解析 XML、过滤无效内容并生成持久化回灌队列；
- `CommentPublisher`：生成 SC/上舰索引评论、楼中楼回复和置顶操作；
- `DanmakuPublisher`：按账号串行回灌并执行限速、退避和熔断；
- FastAPI 路由与 Angular 页面：提供账号、房间投稿规则和任务管理能力。

所有 B 站调用收敛到账号感知的客户端层，业务模块不自行拼装 Cookie、签名或设备请求头。客户端将匿名读取、账号读取、上传和账号写入分开，便于统一限速、审计和测试。

## 5. 持久化模型

SQLite 数据库放在现有持久化配置卷中，启用 WAL、外键和显式迁移。首次升级只创建数据库和迁移记录；所有既有房间的自动投稿、自动评论和弹幕回灌默认关闭，必须在灰度配置中显式启用。主要表为：

- `bili_accounts`：账号 UID、显示名、加密凭据、稳定设备标识、有效期、状态、暂停截止时间和最近错误；
- `room_upload_policies`：房间到账号的映射、投稿开关、标题/简介模板、分区、标签、封面和可见性，以及评论、回灌和过滤选项；
- `recording_sessions`：录制会话、房间、开始/结束时间和聚合状态；
- `upload_jobs`：会话、账号、元数据快照、投稿阶段、重试状态、AID、BVID 和审核结果；
- `upload_parts`：分 P 顺序、视频/XML 路径、文件身份、上传会话、远端文件名和 CID；
- `comment_items`：评论顺序、内容哈希、内容、RPID、置顶状态和错误；
- `danmaku_items`：分 P、CID、来源类型、原始序号、播放进度、文本、样式、优先级、发送状态、次数和错误。

投稿任务在创建时保存房间策略快照，后续修改房间设置不会悄悄改变已排队任务。文件身份由规范路径、大小和修改时间生成，不为大型录像额外计算全文件 SHA-256。数据库唯一约束阻止同一录制会话重复投稿；弹幕唯一键使用 XML 文件身份、原始序号和目标 CID，既能安全重试，又不会误删不同观众同时发送的相同文本。

账号敏感字段使用主密钥加密后入库。主密钥通过 Docker secret 或 `BLREC_CREDENTIAL_KEY` 注入，不写入 SQLite、配置文件、日志或 API 响应。未配置密钥时不能启用账号写入功能。

## 6. 账号与现有读取链路

管理页支持 TV 扫码登录，保存 `access_token`、`refresh_token`、必要 Cookie、CSRF 和账号稳定的 BUVID/设备标识。设备标识只在首次建立账号时生成，刷新凭据时不得重置。应用启动时检查一次账号，此后每 24 小时检查有效期；距离过期不足 72 小时时执行正常刷新。刷新只尝试一次，失败后暂停账号并要求重新扫码。

账号凭据不是降低风控的手段，只在必须登录的操作中使用：投稿、评论、弹幕回灌、登录态直播弹幕连接，以及匿名结果确实异常时的单房间兜底。所有房间的常规批量开播状态继续匿名查询，不把 token 注入每轮批量请求。兜底请求按房间去重，同一房间 10 分钟内最多执行一次，不能演变为 58 个房间逐个高频查询。现有全局浏览器 Cookie 配置暂时保留兼容，但新模块内部使用结构化账号凭据，禁止直接混用 Java 项目的 `name:value` Cookie 格式。

系统允许配置多个真实投稿账号，每个房间固定选择一个账号。同一稿件的评论和弹幕必须由该稿件的投稿账号处理，不使用其他账号分摊队列。

## 7. 投稿状态流

主状态流为：

`录制结束 → 文件稳定 → 创建任务 → 分 P 上传 → 提交投稿 → 等待审核 → 解析 XML → 自动评论 → 弹幕回灌 → 完成`

`UploadCoordinator` 在 `RecordingStartedEvent` 时创建持久化会话，将后续完成的视频和 XML 文件绑定到该会话。只有收到视频完成事件，且文件存在并连续 30 秒保持大小和修改时间稳定，才允许进入上传。录制结束时按照完成顺序形成分 P；XML 缺失不阻止视频投稿，但相应分 P 的回灌会明确标记为跳过。

首期全局一次只运行一个上传任务，避免多个账号同时读取大型录像并占满 NAS 磁盘或上行带宽。上传采用预上传、分块、完成和投稿四个阶段，保存服务端上传会话，使瞬时网络错误只重传失败分块或分 P。提交成功后保存 AID/BVID；审核同步获得每个分 P 的 CID。投稿已成功时，任何评论或回灌失败都不得触发再次投稿。

`ReviewWatcher` 每 15 分钟按账号批量获取待审核稿件，而不是逐稿件持续轮询。审核通过后启动后处理；审核拒绝则保存原因并停止评论及回灌。所有阶段支持暂停、继续、跳过和人工重试。首期不会因任务成功自动删除或移动原始文件。

## 8. 自动评论

自动评论不是 AI 回复或宣传文案。`DanmakuImporter` 从每个分 P 的 XML 读取 `/i/sc` 和 `/i/guard`，按分 P 与播放时间排序，生成如下固定格式：

```text
SC 和上舰列表
1#00:12:34  用户A发送了30元留言：……
2#00:03:05  用户B开通了舰长
```

内容达到 1000 字符安全阈值时拆分：第一段为根评论，其余段落依次作为该评论的回复，并尝试置顶根评论。每一段发送前检查内容哈希和已保存 RPID，容器重启或人工重试不能重复发布。置顶失败只重试置顶，不重发根评论。没有 SC 或上舰时记录为“无内容，已跳过”，不发布空评论。

## 9. 弹幕回灌

回灌是向投稿视频写入原生 B 站弹幕，不把文字压制进视频。每个分 P 的 XML 对应其 CID，普通弹幕保留原始播放进度，并在接口支持范围内保留模式、字号和颜色。SC 与上舰转换为包含用户名和金额/舰长信息的顶部文字弹幕，设为高优先级，但不伪装成真正的付费 SC 样式。

默认过滤规则为：

- 排除抽奖触发、已识别的系统消息和房间黑名单；
- 只去除 XML 中同一原始事件的重复记录，不按文本全局去重，避免删除真实的重复互动；
- 用户等级和粉丝勋章过滤作为房间可选项，默认关闭；XML 缺少相关原始字段时不据此丢弃消息；
- 接口明确拒绝的超长、非法时间或不支持内容记录为跳过，并保留原因。

通过过滤的弹幕全部进入持久化队列，不设置每日上限，也不设置每个分 P 的数量上限。数据库每次读取 500 条仅用于分页和控制内存，不代表发送配额。高优先级消息先处理，普通消息随后按录制时间处理。

每个账号的评论和弹幕写入严格串行。弹幕默认每成功一条至少间隔 25 秒，约为每账号每小时 144 条；管理页展示剩余数量、当前速度和预计完成时间。该速度不能保证 B 站不会风控，只是沿用参考项目的保守基线。回灌后的实际发送者是投稿账号，无法保留直播观众的 B 站发送者身份。

## 10. 风控、错误与恢复

错误处理分为：

- 网络超时和服务端 `5xx`：指数退避并有限重试；
- token 过期：执行一次正常刷新，失败后暂停账号并提示扫码；
- `36703` 等频率限制：逐级延长等待时间，连续触发后熔断账号写操作；
- `36715` 当日操作上限：保留当前弹幕，暂停至下一个北京时间自然日后继续；
- `36704` 稿件未审核：返回审核等待状态，不消耗该条弹幕；
- 内容、时间或权限永久错误：该条标记为跳过或失败，不盲目重复；
- 稿件拒绝、文件缺失或本地文件变化：停止相关阶段并要求人工处理。

应用启动时，过期的执行租约恢复为待执行；每个外部写操作完成后立即提交结果。账号熔断会暂停该账号的上传、评论和回灌，但不会停止录制及匿名状态查询。系统不自动更换账号、代理、token 或设备身份来绕过限制。

提供三个彼此独立的全局紧急开关：停止自动投稿、停止自动评论、停止弹幕回灌。关闭投稿不会停止录制；已提交稿件的阶段状态会保留。日志、异常和页面统一脱敏 Cookie、access token、refresh token 与 CSRF。

## 11. 管理 API 与页面

Angular 前端增加三个入口：

- 账号管理：扫码、登录状态、有效期、刷新/重新登录、账号暂停原因；
- 房间投稿配置：账号、投稿元数据模板、自动投稿、自动评论、弹幕回灌及过滤选项；
- 上传任务中心：会话和分 P、上传/审核状态、BVID 链接、评论预览、回灌成功/待处理/失败数量、预计耗时及最近错误。

任务详情支持暂停、继续、重试和跳过当前后处理阶段。敏感凭据永不返回前端；二维码登录状态使用短期会话标识。首期只提供必要的列表和表单，不增加 AI 编辑、复杂工作流编排或视频文件管理器。

## 12. 测试策略

后端新增 `pytest` 与异步测试支持。自动化测试不访问真实 B 站写接口，而使用本地假服务覆盖：

- TV 扫码、凭据刷新、脱敏和稳定设备标识；
- XML 中普通弹幕、SC、上舰、多 P、时间点与过滤规则；
- 评论拆分、楼中楼、置顶和幂等；
- 分块上传、断点恢复、多 P 投稿、审核与 AID/BVID/CID 保存；
- SQLite 迁移、唯一约束、任务租约和容器重启恢复；
- `36703`、`36704`、`36715`、token 过期、超时及 `5xx`；
- 全局开关关闭写入时录制链路仍可运行。

Angular 使用现有 Jasmine/Karma 测试账号表单、房间配置、任务状态和凭据不泄漏。Docker 冒烟测试覆盖全新数据卷、旧配置升级、持久化卷重启和健康检查。既有格式、静态、类型及前端测试必须通过，新增功能不能改变原有录制和房间批量状态查询结果。

## 13. 线上灰度与验收

自动化测试通过后，在 NAS 上直接选择 3～5 个房间作为首批，不先扩展到全部 58 个房间。首批至少连续运行 3 天，并且每个房间至少完成一场完整直播；尚未开播的房间继续观察。期间验证：

- 开播识别、录制开始/结束、文件可播放和 XML 完整；
- 一场直播只创建一个多 P 投稿，不漏 P、不重复投稿；
- 审核通过后只生成一次索引评论，弹幕进入正确 CID 和时间点；
- 暂停、容器重启和网络中断后可以续跑且不重复写入；
- 上传的 CPU、内存、磁盘 I/O 和带宽占用不造成其他房间漏录；
- 风控响应能够暂停并恢复，不出现持续性账号异常；
- 队列数量、成功率、预计耗时和最近错误可在页面核对。

首批没有漏录、损坏、重复投稿、重复评论、不可恢复任务或持续性风控后，启用全部房间。全量接入仍按房间独立开关，三个全局紧急开关始终保留。

## 14. 完成标准

设计实现完成必须同时满足：

1. 纯 Python 单镜像可在现有 Docker 部署方式中升级并保留旧设置；
2. 扫码账号跨重启可用，正常刷新失败时安全停机而非反复请求；
3. 录制完成可生成幂等的多 P 投稿任务并在网络中断后恢复；
4. 审核、自动评论与弹幕回灌状态彼此独立，失败不会重复投稿；
5. 自动评论只包含 SC/上舰索引且最多发布一次；
6. 所有通过过滤的弹幕均进入队列，不因软件数量上限被丢弃；
7. 风控错误按规则退避或熔断，不轮换身份规避；
8. 自动化、Docker 冒烟和 3～5 房间线上灰度均达到上述验收标准。
