# BLREC 批量直播状态与按需弹幕连接设计

## 1. 背景与目标

当前每个启用监控的房间都会启动独立弹幕 WebSocket，并另外每约 600 秒执行一次单房间 HTTP 状态查询。58 个离线房间仍会保持 58 条 WSS，约每 30 秒各发一次心跳；冷启动和 API 回退也会按房间放大请求。

本设计将状态发现集中到一个匿名批量协调器，只有确认正在直播的房间才建立弹幕 WSS 和取流录制。目标是：

- 离线房间不维持弹幕连接；
- 所有任务共享批量状态快照，不重复查询；
- 批量接口异常时不扩散为 58 路单房间重试；
- 已经开始的录制不因协调器短暂失败而停止；
- 不为常规状态查询携带账号 token/Cookie。

## 2. 方案选择

考虑过三种方案：

1. 匿名批量状态查询，直播时才开 WSS；
2. 使用登录账号的关注直播聚合列表，再低频校准；
3. B 站官方直播开放平台 server 模式。

方案 2 依赖 Cookie 和关注关系，反而增加账号归因；方案 3 需要主播授权、项目审核及 B 站开通 server 模式，不能监控任意公开房间。首期选择方案 1。若未来全部主播完成官方授权，可单独评估迁移到方案 3。

批量接口来自社区逆向资料，不是官方稳定契约。协议失效时系统进入降级状态，不能用账号、代理或多端点并发绕过。

## 3. 组件与职责

- `LiveStatusCoordinator`：维护房间注册、UID 映射、批量轮询、快照和状态机；
- `BatchStatusClient`：只负责匿名批量请求和响应校验；
- `LiveStatusSubscription`：每个 `RecordTask` 接收自己的状态变化；
- `LiveConnectionController`：仅在确认 `LIVE` 时启动现有 `DanmakuClient`、取流和录制，结束时关闭；
- `StatusCircuitBreaker`：对全局限流、响应缺项和连续失败统一退避。

现有 `LiveMonitor` 仍处理直播中的 WSS 事件、房间变更和流可用性，但不再为全部离线房间常驻 WSS，也不再各自执行周期性 HTTP 轮询。

## 4. 房间映射与批量请求

批量主接口采用 `Room/get_status_info_by_uids`，匿名提交 UID 集合并读取 `room_id`、`live_status` 和 `live_time`。房间到 UID 的映射优先复用 `Live.init()` 已获得的用户信息；缺失或房间重定向时，再通过支持多个 `room_ids` 的基础信息请求重建映射。

社区文档没有承诺批量容量，因此首期规则为：

- 默认每批最多 29 个 UID；
- 启动时先用一个小批 canary 验证响应结构，再放开全部批次；
- 批次失败时降低批量大小，不立即改为所有房间逐个查询；
- 输入 UID 去重，多个短房间号映射到同一真实房间时只保留一个事实源。

默认轮询周期 30 秒，可配置到 30～60 秒。58 个房间按 29 个一批时，每轮两次 HTTP：30 秒周期约 5,760 次/日，60 秒周期约 2,880 次/日。主要收益来自离线 WSS 从 58 条降至 0，而不是把 HTTP 请求伪装成零成本。

## 5. 状态机与确认规则

协调器内部状态为 `UNKNOWN`、`STALE`、`PREPARING`、`LIVE` 和 `ROUND`，每个快照保存 `observed_at`、来源和直播开始时间。

- 首次批量 `LIVE`：执行一次目标房间确认，然后启动 WSS 和录制；
- 已直播房间继续 `LIVE`：只更新时间，不重复触发开播；
- 下播：要求连续两次批量非 `LIVE`，或 WSS `PREPARING` 加一次 HTTP 确认；
- 响应缺项、解析失败或超时：标记 `STALE/UNKNOWN`，不得解释为下播；
- `live_time` 变化：作为新一场直播的去重依据，防止重试重复触发；
- 已在录制时，协调器故障只使状态变旧，不能主动停止录制。

开播去重键使用 `(uid, live_time)`。如果上游未返回可信 `live_time`，协调器生成本地 observation key，并要求单房间确认后触发。

## 6. WSS 生命周期

`RecordTask.enable_monitor()` 改为向协调器注册，不直接启动 `DanmakuClient`。状态确认直播后：

1. 建立该房间弹幕 WSS；
2. 获取流信息并启动现有录制流程；
3. WSS 心跳和消息只在直播期间存在；
4. 确认下播后停止录制并关闭 WSS；
5. 任务仍保持注册，等待下一次批量开播信号。

用户主动停止任务或应用关闭时直接注销并关闭连接，不等待下一轮批量状态。直播中 WSS 断线按有界退避重连；超过阈值后依赖批量状态维持 `LIVE`，但不能无限快速重连。

## 7. 匿名优先与有限兜底

常规批量请求不携带账号凭据。只有以下场景允许单房间兜底：

- 批量结果首次报告从非直播切换为直播；
- 某个已直播房间连续缺失；
- 房间号与 UID 映射冲突。

同一房间 10 分钟内最多一次兜底请求，并使用 singleflight 合并并发。默认仍为匿名 Web 请求；只有匿名结果明确受限且房间配置允许时，才可使用投稿子系统中该房间的结构化 Web 会话。账号兜底不能扩展到整批，也不能把 token 放入常规状态轮询。

## 8. 熔断与降级

以下情况触发协调器级熔断：HTTP 429、`-352`、`-412`、连续超时、响应大面积缺项或结构变化。

- 熔断时保留最后快照，已直播 WSS 与录制继续；
- 指数退避后只放行一个小批 canary；
- canary 成功后逐批恢复，不能同时释放全部 58 个房间；
- 连续协议错误进入人工暂停并在页面提示；
- 禁止同时调用多个旧接口、HTML 页面或账号池来“碰运气”。

若批量接口长期失效，可由全局开关临时恢复旧单房间监控，但必须明确显示请求成本；两种模式不能同时运行。

## 9. 可观测性与管理

管理页面显示：当前模式、轮询周期、批次数、最近成功时间、快照年龄、批量缺项、熔断状态、活跃 WSS 数、开播发现延迟和单房间兜底次数。

指标要求：

- 离线 WSS 数始终为 0；
- 活跃 WSS 数约等于同时直播房间数；
- 单轮每个房间只出现一次状态结果；
- 批量错误不会造成单房间请求突增；
- 状态缺失不会触发错误下播。

## 10. 测试策略

自动化测试覆盖：

- 58 个房间分批、去重、UID 重映射和响应缺项；
- `UNKNOWN/PREPARING/LIVE/ROUND` 转移与 `(uid, live_time)` 去重；
- 首次直播确认、连续两次下播确认和 WSS 辅助信号；
- 429、`-352`、`-412`、超时、结构漂移和 canary 恢复；
- 熔断期间已录制任务不停止；
- 离线不创建 WSS，直播才创建，结束后释放；
- 单房间兜底 10 分钟冷却和 singleflight；
- 旧模式与批量模式互斥。

假服务模拟批量响应和 WSS，不依赖真实 B 站。线上灰度前执行匿名只读 canary，确认批量端点结构和批次大小仍有效。

## 11. 灰度与完成标准

与投稿系统相同，首批选择 3～5 个房间运行至少 3 天且每个房间覆盖一场直播，然后扩到全部房间。灰度期间可对选定房间做低频、匿名的旧 HTTP 路径抽样对照，但旧监控不得建立额外 WSS、触发第二份录制或与批量协调器同时驱动状态机。

完成标准：

1. 离线房间不建立 WSS；
2. 开播发现延迟符合 30～60 秒配置；
3. 一场直播只触发一次开始和结束；
4. 批量缺项、限流或结构错误不会造成误下播和请求风暴；
5. 已开始的录制在协调器短暂故障时继续；
6. 常规状态查询保持匿名，账号凭据只用于显式允许的单房间兜底；
7. 58 房间请求量、活跃 WSS 数和错误均可观测。
