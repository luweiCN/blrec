# 录像保留与容量管理设计

## 目标

把现有“磁盘空间不足时删除旧录播”与房间级录像删除规则合并为一套可追踪的清理机制。全局设置决定何时产生容量压力，房间策略决定哪些录像可以删除。只删除视频媒体，永久保留弹幕 XML、封面、元数据和数据库记录。

## 房间删除策略

每个录制房间在投稿设置中选择一种“本地录像保留”策略：

- 从不删除
- 容量超限时删除
- 录制结束后 N 天删除
- 上传完成后 N 天删除
- 投稿成功后 N 天删除
- 审核通过后 N 天删除

除“从不删除”和“容量超限时删除”外，延迟天数为非负整数，`0` 表示触发条件满足后尽快清理。上传完成按每个分 P 已成功完成远端上传计算；投稿成功按 B 站接受稿件并返回 AID/BVID 计算；审核通过按系统核对稿件和分 P 成功计算。

删除策略使用房间当前设置，并立即作用于该房间所有仍存在的历史录像和之后的新录像。没有删除策略的既有数据在迁移时设为“投稿成功后 5 天删除”，新建房间也使用相同默认值。切换策略后，已经满足触发条件和延迟时间的历史视频会进入最近一次清理周期；保存前显示将立即到期的录像数量和总大小，并要求确认，避免误操作。

“容量超限时删除”是独立的房间策略，不是全局强制覆盖。选择其他策略的房间不会因为全局容量或磁盘空间不足而被提前删除；“从不删除”受到绝对保护。

## 全局容量设置与预警

设置页的磁盘空间模块保留检查周期和物理磁盘最低剩余空间，并增加：

- 应用录像容量上限，例如 500 GB；关闭时不按应用容量触发清理。
- 容量预警余量，例如 20 GB，必须小于容量上限。
- 当前受管视频占用、容量剩余量和物理磁盘剩余量。

受管容量只统计数据库登记且仍存在的录像媒体；同一路径只计算一次。FLV、MP4、TS 等视频及 HLS 自有媒体片段属于录像容量，XML、JSON、封面、日志、数据库和其他目录文件不计入。

以上限 500 GB、预警余量 20 GB 为例：受管视频达到 480 GB 后进入预警，但不删除；超过 500 GB 后触发容量清理。预警在全局页面提示和设置页容量状态中持续显示，同时写入结构化警告日志。本期不发送邮件、微信等外部通知，后续异常通知功能可直接订阅相同状态。

现有“空间不足删除旧录播文件”开关移除，由房间的“容量超限时删除”策略取代。物理磁盘剩余空间低于配置阈值时，也触发同一个容量清理流程。

## 清理流程

容量清理只查询选择了“容量超限时删除”的房间，按录像结束时间从旧到新处理。应用容量超限时清理到不再超出上限；物理磁盘空间不足时清理到重新高于最低剩余空间。达到预警线但尚未超限时不清理。

候选文件必须已经结束写入和后处理，且当前没有上传进程持有。存在尚待上传的任务时先跳过，避免为释放空间破坏承诺中的投稿；上传完成或任务取消后可再次成为候选。没有足够安全候选时进入“容量清理受阻”状态并保留警告，不越权删除其他房间或活动文件。

状态触发型策略由同一个清理执行器处理。每次删除前重新核对数据库状态、规范化路径和普通文件类型。只删除该分 P 登记的源视频、成品视频或属于其 HLS 媒体集合的文件；源文件与成品文件路径相同时只删除一次，绝不根据相同文件名前缀顺带删除 XML 或图片。

删除成功后保留分 P 记录并写入删除时间、触发原因和释放字节数，以便列表显示“本地录像已删除”及决定是否跳转 B 站。文件本来已不存在视为幂等完成；权限或占用错误记录失败原因并在后续周期重试，不能像参考 Java 实现那样在删除失败后仍标记成功。

## 与录制、上传和播放的关系

清理执行器单实例串行运行，并与录制、后处理和上传状态联锁。正在录制、生成成品、计算文件身份或上传的文件一律跳过。重启后根据持久化事件时间重新计算到期任务，不依赖内存定时器。

本地录像删除后，弹幕仍可查看，自动评论与弹幕回灌仍可读取 XML。审核通过且有 BVID 的分 P 改为跳转对应 B 站分 P；尚未投稿或未审核通过时只显示本地文件已删除，不构造不可用链接。

## 验证

测试先覆盖预警边界、容量超限、物理磁盘不足、房间策略隔离、默认“投稿成功后 5 天删除”的数据迁移、策略变更立即作用于历史录像、按时间排序、延迟 0 天与 N 天、重启恢复、重复路径计数、删除幂等和失败重试。并验证任何清理都不会删除 XML、封面、正在录制、后处理、上传或等待上传的文件。

集成测试构造小容量目录，确认只回收选择“容量超限时删除”的最旧安全视频；候选不足时报告受阻。前端测试覆盖容量展示、预警、受阻提示、策略动态字段，以及删除后本地播放入口切换为正确的 B 站分 P 链接。
