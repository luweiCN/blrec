# 录制场次与上传任务生命周期设计

## 目标

上传任务页面以“录制场次”为一行，上传任务只是场次可能拥有的子流程。没有上传任务的场次也必须能够决定是否上传、手动创建任务、批量操作和本地删除，不能成为只能占用容量的死记录。

页面只展示一个面向用户的主状态。录制、文件处理、分 P 上传、投稿提交、审核、评论、合集和弹幕回灌的内部状态放入详情，不再要求用户理解多套状态机。

本设计取代既有设计中“活动任务不能删除”“启用投稿规则可作用于历史录像”以及“操作必须依赖上传任务 ID”的约束。

## 领域边界

- **录制房间**是长期配置。
- **录制场次**是一次开播及其所有分 P、弹幕和本地文件。
- **上传意图**是某一场是否准备投稿的场次级决定。
- **上传任务**只在场次计划上传且文件就绪后创建。
- **投稿规则**只为未来场次提供默认上传意图和投稿设置。

所有列表操作以场次 ID 为入口。存在上传任务时，后端再定位对应任务；不存在上传任务时，仍可直接操作场次。

## 投稿规则与本场覆盖

投稿规则从启用时刻开始生效，只适用于此后开始的录制场次。直播已经开始后才启用规则，当前场不自动投稿，从下一场开始。关闭后重新启用会产生新的生效时刻；启用期间修改模板不会把历史场次纳入上传范围。

场次开始时冻结当时的默认上传意图及投稿设置。之后修改房间规则不会改变已经开始或已经结束的场次，也不会自动为历史场次补建任务。

录制期间允许在以下两个状态之间切换：

- **本场上传**：仅让当前场在结束后创建上传任务。
- **本场不上传**：仅阻止当前场创建上传任务。

切换只影响当前场，不修改房间长期投稿规则。没有可用投稿设置时选择“本场上传”，先打开本场投稿设置；保存后才确认上传意图。上传已经开始后不再切换上传意图，用户可通过“删除”立即终止整个本地流程。

历史场次只能由用户明确点击“创建上传任务”。该操作只处理选中的场次，并允许先确认本场投稿设置和账号，绝不因为房间规则启用而连带处理其他历史场次。

## 用户状态与操作

“详情、播放现有本地文件、查看弹幕”在内容存在时始终可用。“删除”在所有状态下始终可用，其具体终止语义见下一节。

| 主状态 | 含义 | 状态专属操作 |
| --- | --- | --- |
| 录制中（结束后上传） | 当前场继承规则或被手动设为上传 | 本场不上传 |
| 录制中（未安排上传） | 当前场没有上传意图 | 本场上传 |
| 文件处理中 | 已结束录制，正在归集或后处理 | 本场上传、本场不上传 |
| 文件处理失败 | 分 P 处理失败、缺失或无法恢复 | 重试处理、重新扫描 |
| 未安排上传 | 文件就绪但本场没有上传意图 | 创建上传任务、设为不上传 |
| 已设为不上传 | 用户明确跳过该场 | 创建上传任务 |
| 待上传 | 上传任务已创建，尚未发送数据 | 立即上传、修改本场投稿设置、切换账号、设为不上传 |
| 上传中 | 正在上传一个或多个分 P | 暂停上传 |
| 上传已暂停 | 用户暂停、账号不可用或发生可安全重试的错误 | 继续上传、重试；尚无远端结果时可改账号和设置 |
| 投稿中 | 分 P 已上传，正在提交稿件 | 无破坏性专属操作 |
| 投稿结果待核对 | 投稿请求已发出但结果不确定 | 核对 B 站状态，禁止盲目重试 |
| 等待审核 | 已取得 AID/BVID，等待 B 站审核 | 立即检查审核状态 |
| 审核未通过 | B 站明确拒绝稿件 | 查看原因、修改后重新提交原稿、重新投稿为新稿件 |
| 审核通过（后续处理中） | 稿件可见，合集、评论或弹幕回灌仍在执行 | 打开稿件、重试失败的后续步骤、检查并修复转码、重新投稿 |
| 处理完成 | 审核及所有启用的后续步骤完成 | 打开稿件、检查并修复转码、重新投稿 |
| 转码修复中 | 正在核对、重传或更新转码失败分 P | 查看修复进度 |
| 正在删除 | 已接受删除，正在终止关联工作并释放文件 | 无，等待完成 |
| 删除失败 | 某个工作未能停止或文件无法删除 | 重试删除、查看原因 |

除详情、播放和查看弹幕外，所有适用操作都支持批量执行。混合状态批量操作先展示可执行、将跳过和失败的项目；用户确认后逐条重新校验，单条失败不阻断其他场次。

## 任意状态下删除

删除表示“放弃本系统对该录制场次的一切后续工作”，不是 B 站删稿。操作采用持久化的两阶段流程：

1. 先把场次标记为“正在删除”，记录终止标记，阻止任何 worker 领取新工作或重建上传任务。
2. 取消并等待该场次关联的录制、后处理、上传、投稿、审核检查、转码修复、合集、评论和弹幕回灌工作释放资源。
3. 文件句柄和执行租约释放后，删除归属于该场次的本地视频、弹幕、封面、临时文件及数据库记录。
4. 全部成功后该行从列表消失；失败则保留“删除失败”和具体原因，可再次重试。

删除正在录制的场次时，停止当前场录制并忽略该次直播余下时间，但保留录制房间配置和监控；主播下一次开播仍按房间设置正常工作。这样可防止当前直播仍在线时，后台马上重新创建刚删除的场次。

删除正在上传或处理的场次时，停止后续本地请求，但无法撤回已经被 B 站接收的分片、评论、弹幕或投稿请求。删除处于投稿中、结果待核对、等待审核、审核通过或处理完成的场次时，确认弹窗必须明确提示：B 站可能已经存在稿件，本操作不会删除或修改它。

后端不得在本地删除流程中调用 B 站删稿接口。对于已经发出的外部请求，只保证删除确认后不再发出新的后续请求。

## 容量与保留

未安排上传和已设为不上传的场次同样属于受管录像，必须参与容量统计和房间保留策略。它们没有“投稿成功”里程碑，因此状态触发型策略不会假装已经投稿；用户可以手动删除，选择“容量超限时删除”的房间则可在文件停止写入后成为容量清理候选。

自动容量清理仍不得抢占活动文件。与用户主动删除不同，自动清理遇到录制、处理或上传中的场次只会跳过，不会主动取消任务。

## 内容入口与稿件链接

本地录像播放和弹幕查看是两个独立功能。分 P 的“文件”列只负责打开本地播放器，“弹幕”列只负责打开弹幕查看器；两个弹窗不使用标签页互相切换。

录制中的 FLV 采用打开时固定的播放快照，显示当时总时长并可在当时已录范围内拖动。快照复用录制器实时维护的关键帧索引，通过虚拟 FLV 元数据和 HTTP Range 定位，不转码、不复制整个录像，也不影响录制继续写入。重新打开播放器才读取后来新增的内容。

稿件审核通过并保存 BVID 后，场次标题显示为蓝色稿件链接。审核流程逐个核对远端分 P 并回填 CID 后，详情中的 `P1`、`P2` 等也显示为对应 `?p=` 直达链接。本地文件仍存在时同时保留“播放”和 B 站链接；本地文件删除后 B 站链接继续可用。

## 实现边界

录制场次需要持久化上传意图、冻结的投稿设置版本、删除状态和删除失败原因。投稿规则需要独立的生效时间，不能复用每次编辑都会变化的更新时间。

所有 worker 在领取和提交阶段都检查场次终止标记。删除动作先写终止标记，再发送取消信号；不能先删数据库再依赖内存任务自行停止。活动录制还需要为当前直播保存忽略标记，直到检测到下播，避免同一场被重新识别。

上传意图切换、创建上传任务、暂停、继续、重试和删除使用同一套场次操作 API，并返回每条场次当前可用动作及不可用原因。前端只渲染后端能力，不复制状态判断。

## 备选方案与取舍

仅给投稿规则增加生效时间的最小补丁可以阻止历史补建，但无法解决未创建任务无操作、双状态展示和死文件问题，因此不采用。

开播时直接创建空上传任务可以消除“未创建任务”，但会把录制、文件处理和上传状态耦合成一个对象，并产生大量最终不投稿的空任务，因此不采用。

采用“场次为主、上传任务为子流程”可以统一状态和操作，同时保留上传状态机的幂等边界。

## 验证

自动化测试至少覆盖：规则首次启用不追溯、直播中启用不包含当前场、关闭后重新启用重新计算生效时间、编辑规则不改变既有场次、本场上传与不上传双向切换、历史场次仅单条手动建任务，以及未创建任务可删除和参与容量候选。

删除测试覆盖每一种主状态，重点验证活动录制不会重建、后处理和上传被取消、worker 在终止标记后不再发请求、进程重启继续未完成删除、文件占用失败可重试、混合批量删除部分成功，以及所有路径都不调用 B 站删稿接口。

前端测试覆盖单一主状态列、后端能力驱动按钮、本场上传/不上传切换、所有状态均显示删除、不同阶段的删除风险文案、正在删除/删除失败反馈和批量预览，以及播放/弹幕入口分离、录制快照拖动、稿件标题链接和逐分 P 直达链接。
