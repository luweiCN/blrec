# 本轮调查与设计计划

## 目标

查明当前 Python 实现相较 `biliupforjava` 出现预上传 HTTP 406、投稿业务码 21588 的原因；定位高光片段自动创建上传任务及弹幕偏少问题；在证据明确后设计片段资产库、弹幕指标和全应用一致的信息密度方案。

## 成功标准

- 用代码位置和真实任务数据说明 Java/Python 上传协议、重试、并发和投稿载荷差异。
- 对任务 17、18、21 的失败阶段给出可复核结论，不猜测未保存的 B 站错误文案。
- 解释高光片段 2、3 为何在创建时生成暂停的上传任务，并明确解耦边界。
- 对今晚至少一场新旧服务同时采集的直播，比较 XML 条数、文件时间范围、WebSocket 连接/重连日志，定位弹幕减少发生在哪一层。
- 记录片段永久存储、独立容量、片段列表、弹幕统计和全局视觉调整需求，形成待确认设计。

## 阶段

1. **已完成**：保存需求和调查假设。
2. **已完成**：调查 `biliupforjava` 的预上传、重试、投稿载荷与错误处理。
3. **已完成**：追踪高光片段创建与上传任务创建调用链。
4. **已完成**：从 NAS 新旧服务、XML、数据库和日志对比弹幕采集。
5. **已完成**：按已确认方案实现动态上传、弹幕连接、片段库和列表重排。
6. **进行中**：发布 beta.16、更新 NAS，并用真实失败任务验证恢复机制。

## 本轮已确认需求

- 新增“片段管理”侧边栏页面，显示名称、时长、主播、来源场次、大小、预览、下载、投稿、删除及投稿进度。
- 片段存入与 `rec` 同级的独立目录，永久保存，不计入录制容量额度，不参与自动清理；真实磁盘剩余空间仍须反映其占用。
- 高光片段创建不得自动创建上传任务；只有用户明确执行投稿时才创建。
- 上传任务列表增加弹幕总数、成功发送数等状态。
- 上传任务、录制任务采用统一的复合列表行：控制在少量核心列内，不把横向滚动作为常规交互；相关数据合并为两到三行的复合单元格，以适度增加行高换取更高的信息容量。
- 复合单元格按业务归组，不把每个字段都拆成独立列。例如“录像”合并分 P 数、时长和大小，“上传”合并阶段、进度和速度，“弹幕”合并采集总数、已回灌、待处理和失败数。
- 上传任务、录制任务的信息密度和响应式布局需调整，并基于统一表格规范复用到其他数据列表，不能只修两个页面。
- 房间管理列表的“自动投稿”单元格升级为投稿摘要：始终显示自动投稿开关、实际投稿账号、可见范围和稿件类型（转载/自制）；显示立即/定时发布以及本地视频删除策略；配置了合集时显示合集名称，配置不可用时显示醒目的原因摘要。弹幕回灌关闭时额外显示例外标记。
- 房间管理的“监控与录制”单元格在配置了标题关键词时显示“标题匹配”摘要，避免用户只看到直播中却不知道为何没有开始录制。关键词较多时只显示数量和前两项，完整内容悬浮查看。
- “仅自己可见”使用醒目标记，“公开”使用普通文字；即使自动投稿暂时关闭，也保留灰显的投稿摘要，便于用户在重新开启前核对配置。
- 标题/简介/分 P 模板、标签、封面、评论细项等低频配置继续留在投稿设置弹窗，不占用房间管理主列表。
- 批量重试不得混入人工暂停任务；B 站错误需保留安全的接口阶段、错误码和文字。
- 上传改为按分 P 调度的动态队列：从低容量开始，成功后逐步放行；收到 HTTP 406 时停止创建新的预上传会话，保留已在上传的分 P，冷却后自动恢复。
- 自动投稿前过滤真实媒体时长不足 60 秒的小分片，避免异常断流碎片进入 B 站稿件；具体是仅排除投稿还是同时删除本地文件仍待确认。
- 弹幕 WebSocket 改为登录态优先、匿名兜底。登录 Cookie 无法自动恢复时产生账号失效事件；配置通知渠道后发送通知，未配置时保留页面状态和日志。
- 动态上传窗口第一版允许增长到 5，不使用 3 作为上限；窗口仍由成功逐步增加、遇到 406 收缩。长期是否取消内部安全上限由线上数据决定。
- 分 P 封闭后立即进入预上传队列，不等待整场直播结束；直播结束时只需上传最后尚未完成的分 P，再创建稿件。
- 修复录像详情中的弹幕列表无法显示问题，并把视频与弹幕放入同一播放体验：视频下方为固定高度、可滚动弹幕列表；播放时自动高亮当前时间附近的弹幕并跟随滚动，点击弹幕可跳转到对应时间。
- 全面审计自动暂停上传任务的代码路径。可恢复的频控、网络、服务端、进程重启等错误必须自动续跑；仅不可恢复的本地文件、账号、配置或明确内容拒绝允许进入需要处理状态。
- 新机制上线前不重置失败任务。上线后丢弃失败任务已有的 UPOS 会话、远端文件名和中间状态，按新过滤规则重新上传；历史远端对象不复用。

## 已确认产品决策

- 数据列表采用“少量核心列 + 复合单元格分层信息”，桌面宽度下不出现横向滚动；窄屏时改为上下分区的列表卡片，不要求用户左右拖动。本轮不增加可配置列，避免把常用信息藏进设置。
- 正在上传的片段被手动删除时，沿用上传任务既有规则：先取消本地任务再删除片段文件与记录；任何情况下均不删除 B 站已有稿件。
- 不足 60 秒的分 P 保留在录制任务与磁盘，只排除自动投稿。
- 动态上传窗口第一版采用内部安全上限 5；后续根据线上 406 与吞吐数据决定是否调整。

## 建议修复边界（等待确认）

- 预上传：以分 P 为排队单元；动态窗口从 1 开始，完整成功后加 1，不设置固定目标容量，实现层暂用 5 作为安全上限。406 时窗口收缩并进入冷却，已取得会话的分 P继续上传；冷却后从 1 个探测任务恢复。连续 406 使用递增退避，不转成人工暂停。
- 队列错误分级：406、429、5xx、出口故障等系统性问题冻结新建会话；单个文件无视频轨、时长不足或损坏时隔离该分 P，不阻塞其他正常任务。
- 分 P 有效性：投稿前以 ffprobe 校验真实时长/视频轨；不足 1 秒等违反 B 站硬条件的碎片保留在录像记录中，但不进入上传稿件，并记录排除原因。
- BVC：持久化预上传 CID；保存 21588 的 `bvc_check` 安全详情并映射到具体分 P；可修复的媒体错误进入现有 remux/重传流程，内容审核拒绝才暂停并展示明确原因。
- 投稿重试：网络、5xx、频控等瞬时错误自动退避；不盲目重试明确的内容拒绝。Java 的“全部错误最多重投 5 次”仅作为兜底参考。
- 弹幕采集：开播后使用登录态 WebSocket 获取完整消息；Cookie 不可用或登录态握手失败时先尝试自动续期，无法恢复则标记账号异常、按已配置渠道通知并回退匿名。每场记录连接模式、收包数、DANMU_MSG 数与重连次数。
- 弹幕列表单元格：第一行显示采集条数，第二行显示回灌进度（如“已回灌 80 / 124”），仅有异常时显示第三行失败摘要；完整错误留在详情抽屉。
- 列表行保持 4 个内容区：任务身份、录像信息、处理进度、投稿配置，外加勾选框和操作区。开始时间并入任务身份；上传、审核和弹幕回灌并入处理进度；账号、可见性和合集并入投稿配置。
- 片段资产：新增独立 `/clips` 根目录、全局片段列表与投稿进度；不纳入录制容量统计和自动清理，仍占用真实磁盘并参与磁盘剩余空间告警。
- 全局视觉：抽出统一页面壳、工具栏、数据表、状态单元与操作区规范；信息密集页面使用更宽的数据页规格，设置/账号等编辑页面维持阅读宽度。
