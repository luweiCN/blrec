## 初始代码发现

- Python 上传器在预上传前把分 P 状态写成 `preupload`；该调用发生在 UPOS 分块重试循环之外。
- 任意 `BiliApiError` 冒泡到上传任务层后都会把任务设为暂停，HTTP 406 当前没有退避重试。
- 投稿层捕获 `BiliApiError` 时只保存错误码，丢弃了 `public_message` 和 `operation`。
- 批量失败任务查询没有排除 `operator_paused=1`；重新排队逻辑也没有清除该标记，存在人工暂停任务被错误混入的问题。

## Java 运行版本取证

- NAS 正在运行的旧上传容器是 `biliupforjava-blrec`，镜像为 `mwxmmy/biliupforjava:latest`，已连续运行 7 天。
- 该镜像创建于 2025-09-15，摘要为 `sha256:7abc0e...`；容器把 `/volume1/docker/blrec` 挂载到 `/bilirecord`。
- Java 仓库默认分支为 `web_local`。与镜像创建日期对应的代码提交是 `a366e4f`（标签 `1.4.1`，2025-09-15），因此后续以该版本为主要对照，而不是仓库中 2023 年的 `latest` 标签。
- NAS 旧录制服务仍为 `acgnhiki/blrec:latest`，挂载 `/volume1/docker/blrec/rec`、`config` 和 `log`，具备与新服务做同场弹幕差异对比的条件。

## Java 与 Python 预上传差异

- Java 预上传只发送 `r/profile/name/size`；Python 额外发送 `ssl=0`、`version=2.8.12`、`build=2081200`。
- Java 使用 Chrome 103 UA；Python 当前使用 Chrome 63 UA。Java 还把所选上传线路字符串（例如 `?os=upos&zone=cs&upcdn=bldsa`）写入 `Referer`，Python 使用正常的 B 站首页 Referer。
- Java 不检查 HTTP 状态码，直接解析响应体；当解析出的 `OK==0` 时会无限循环，每 10 秒重新请求预上传。Python 遇到任何 HTTP 4xx 会立即抛出 `BiliApiError` 并暂停整条任务。
- Java 分块失败最多累计重试 200 次，每次间隔 10 秒；UPOS 合并最多重试 5 次；部分失败还会重新排队整个分 P。它不是“从不失败”，而是会把大量瞬时失败吞在内部重试里。
- Python 的 HTTP 406 发生在任务 17 已成功预上传 10 个分 P之后，因此不像固定请求格式完全不兼容，更符合短期频控、并发节奏或特定请求上下文触发。

## Java 与 Python 创建稿件差异

- Java 在创建稿件前调用独立接口生成新的 `buvid3`、`buvid4`，并临时追加到提交 Cookie；Python 直接使用账号凭据中已有的 Cookie。
- Java 投稿包含 `build=1088`、`desc_v2`、`dynamic_v2`、`topic_grey`、`web_os`、`human_type2` 等网页投稿字段；Python 当前载荷更精简，同时增加了原创声明等新字段。
- Java 识别简介中的 `${@uid}` 模板并生成结构化 `desc_v2`，同时确保 `@用户名` 后有空格。任务 21 的简介包含普通文本 `@奇游手游加速器`，Python 没有生成 `desc_v2`，这是 21588 的重点候选，但尚需接口资料或可重复验证确认。
- Java 若响应包含“验证码”会等待 120 秒再提交一次；其他投稿失败会增加 `uploadRetryCount`，后台每分钟重新扫描，最多自动尝试约 5 次。Python 对 B 站业务拒绝立即永久暂停。因此用户在 Java 页面上不容易看到短暂拒绝，并不代表 Java 从未收到过拒绝。
- Java 保存并记录完整投稿响应；Python 历史任务只保存数字错误码，导致任务 21 的原始 B 站错误文字已不可恢复。

## Java 线上日志核对

- 过去 7 天 Java 容器记录了 6 次投稿响应，没有记录投稿失败、验证码等待、预上传限流等待、UPOS 合并重试、分 P 重新排队或总上传异常。
- 这说明 Java 的重试机制确实存在，但本周这 6 次投稿没有触发；不能用本周日志证明 406 一定由重试掩盖。
- 下一步需结合 Python 任务 17 的 16 个分 P时间线判断是否是短时间大量预上传触发，并核对 Cookie 中 buvid 组成。

## 406 的现场时间线

- 任务 17 的 16 个分 P并非正常均匀切片：P3–P11 都发生在 22:28:39–22:29:36，很多只有 1–6 秒；P11 仅 61,202 字节。
- 上传阶段在 23:53:32–23:53:46 的 14 秒内连续成功创建并完成了 P3–P10 的 8 个 UPOS 会话；加上前面的 P1、P2，短时间窗口内共成功预上传 10 次。
- 第 11 次预上传（P11）在 23:53:46 发起后收到 HTTP 406；25 秒后同账号的任务 18 首次预上传也立即收到 406。
- 该时间线强力支持“同账号/同出口短时间预上传次数触发 B 站限流”而非文件格式、固定请求字段或单个文件损坏。任务 18 是 2.9 GB 正常文件，排除了“仅因 P11 太小”的解释。
- Java 每 10 秒重试 `OK=0` 的机制正好能跨过这种短时窗口；Python 把第一次 406 永久暂停，因此差异会直接暴露给用户。
- Java 的底层 `HttpClientUtils` 已进一步核实：它把 HTTP 406 响应原样包装成 `HttpClientResult`，调用者完全没有检查状态码；406 JSON 被反序列化成 `PreUploadBean` 后，未出现的整型 `OK` 默认为 0，于是确实进入每 10 秒一次的无限预上传循环。这不是推测，而是该版本处理 406 的实际代码路径。
- 因此 Java“从未遇到 406”的真实含义是：即使遇到，它也不会把 406 暴露成任务失败；只要 B 站最终解除限流，它就自动继续。当前 Python 则把同一个可恢复事件转成了永久人工暂停。

## API 备份文档核对

- 备份文档的较新预上传示例使用普通视频 profile `ugcfx/bup`，并给出 `version=2.14.0.0/build=2140000`；Java 和 Python 都仍使用旧的 `ugcupos/bup`，Python 版本参数也较旧。旧 profile 并非立即失效（任务 17 前 10 次成功），但应作为协议对齐项测试。
- 文档明确列出预上传响应自带 `chunk_retry`、`chunk_retry_delay`、`threads` 和 `timeout`；当前 Python 没有采用服务端返回的重试建议，而 Java 使用自定义的固定重试。
- Web 创建稿件文档把 `web_os`、`desc_format_id`、`recreate` 等列为必要字段；Python 缺少 `web_os`，并使用 `desc_format_id=0`，与文档示例 `9999` 不一致。
- 文档说明简介存在 `@` 时应同时传递结构一致的 `desc_v2`；任务 21 只有普通 `desc`，没有 `desc_v2`。
- 文档没有记录业务码 21588 的含义，仍需搜索历史 issue 或通过安全的验证/日志捕获确认，不能直接把它定性为某一个字段错误。
- 风控文档强调正常 UA、Referer、`b_nut/buvid3/buvid4/bili_ticket` 可降低风险；Java 每次投稿补 buvid3/4，Python 是否完整携带这些 Cookie 仍待核对。

## 21588 已定位

- B 站 2026-07-16 发布的当前投稿前端 bundle 明确处理业务码 `21588`：当响应 `data.bvc_check` 存在时，按其中的 CID 找到对应分 P，并把服务端返回的错误文字写入该视频的 `videoCheckResult`，标记 `service_defender=true`。
- 因此 21588 不是普通简介、原创声明或 Cookie 校验错误，而是 B 站的视频内容/文件检测（前端命名 BVC/service defender）拒绝了一个或多个已上传分 P。
- 任务 21 的四个分 P均有远端文件名，但 Python 没保存响应中的 `data.bvc_check`，所以目前还不知道具体是哪一 P及 B 站给出的错误文字。
- Java 代码只检查是否返回 AID/BVID；失败后保存完整响应并由定时任务重投。它没有针对 21588 修复视频，但完整日志会保留 `bvc_check`，并且多次重投可能掩盖偶发检测失败。
- 当前 B 站前端已使用 `ugcfx/bup`、`autoRetry=true` 和 10 MB 分块；进一步说明 Python 应切换新 profile，并按前端重试语义处理预上传瞬时错误。
- 任务 21 的四个源文件已在 NAS 上用 `ffprobe` 实测：P1 仅 6.267 秒，P2 仅 0.901 秒且画面方向从 1544×1080 突变为 1080×1544，P3/P4 分别约 7198 秒和 2232 秒；四个文件本身均能被 ffprobe 读取。
- B 站当前官方上传器的本地检测明确把错误 12 定义为“该视频时长不足1秒，请检查文件后重新上传”。因此 P2 已经违反官方上传前置条件，极有可能就是任务 21 的 BVC 拒绝分 P；因旧实现未保存 `bvc_check`，只能定为高概率而不能把历史错误精确复原。
- Java 版在发布前会删除小于房间 `durationLimit` 或 `fileSizeLimit` 的分 P；`durationLimit` 默认就是 60 秒。这是它很少把录制断线产生的 0.9 秒异常碎片提交给 B 站、因而很少碰到 21588 的另一项关键机制。当前 Python 没有等价的协议级最低有效性校验。
- 当前 Python 的 `UposSession` 已持有 `biz_id`，但完成分 P 后只保存 `remote_filename`，没有把 `biz_id` 同步为 `upload_parts.cid`。这会让 21588 响应即使保留了 `bvc_check`，也无法直接映射回本地 P；修复需同时补齐 CID 持久化。

## 当前 B 站上传器对 406 的处理

- 2026-07-16 的创作中心上传器仍显式识别 HTTP 406：若响应体业务码也为 406，则展示“上传视频过于频繁，请稍后再试”，证明 406 就是预上传频率限制，而非文件格式错误。
- 当前上传器配置 `autoRetry=true`、`profile=ugcfx/bup`、10 MB 分块，但这里的 `autoRetry` 主要属于分块上传器。预上传请求另有最多 10 次、300 ms 间隔的请求包装器，其 `shouldRetry` 明确排除 HTTP 400–599，因此 406 不会在官方网页内自动重试，而是停止该视频任务并提示用户稍后再试。
- 这意味着修复不能简单照抄“官方 10 次重试”：对于无人值守录播系统，应在应用任务层把 406 识别为可恢复的频控，做较长退避并自动续跑；Java 版隐藏该错误的具体路径仍需以实际 HTTP 客户端行为继续核对。

## 高光片段与暂停上传任务

- 当前 `create_clip` 只插入 `highlight_clips` 并排队生成 MP4/XML，不会创建上传场次或上传任务；只有调用 `/clips/{id}/upload-task` 才会先创建高光上传场次、保存完整投稿设置并创建可执行任务。
- 暂停的任务 2、3 创建于 2026-07-17 08:29/08:31。当时前端旧实现的“投稿设置”按钮实际上直接调用 `createUploadTask(clip.id)`；后端随即创建 `operator_paused=1` 的草稿任务，再打开任务编辑器。这正是用户所说“我只是创建/查看片段，却出现暂停任务”的历史来源。
- 该流程已在 2026-07-17 12:02 的提交 `e2a5639` 改为：先打开投稿设置弹窗，只有弹窗保存事件才把完整设置随 `/upload-task` 一次提交；新任务初始状态为 `ready`，不再人工暂停。当前 beta.15 已包含修复。
- 因此现有任务 2、3 是升级前遗留数据，不代表当前创建片段仍会自动生成任务。后续片段管理页仍需保持这一边界：创建片段与投稿完全解耦；只有明确点击“投稿”并保存设置才创建任务。

## 弹幕明显偏少

- 新数据库的 `danmaku_count` 与对应 XML 实际 `<d>` 节点完全一致，排除了列表统计错误；问题发生在采集端。
- 同一时间、同一房间的旧/新服务构成了直接 A/B 对照。房间 26258444 在约 2 小时 37 分的同场直播中：旧 BLREC XML 共 124 条（87+37），新服务只有 12 条（8+4）。原始 JSONL 也分别收到约 1699 与 181 条消息，说明不是 XML 转换过滤，而是新 WebSocket 收到的上游消息总量就少约一个数量级。
- 房间 24767459 也呈现同方向差异：旧服务同场分片约 29 条弹幕，新服务只有 4 条；后续同场旧服务 7 条、新服务 1 条。不是单个主播恰好没人发言。
- 新服务日志显示匿名 WebSocket 从 23:19:04 到 01:56:38 全程 Auth OK、无重连、无接收错误，因此“连接断掉导致丢失”被排除；B 站接受了匿名握手，但返回的是明显不完整/抽样的消息流。
- 旧 BLREC 的全局 `[header]` 确实配置了完整 Cookie，包含 `DedeUserID`、`SESSDATA`、`LIVE_BUVID`、`buvid3/4` 等；旧代码会把这些 Cookie 直接用于弹幕 WebSocket 身份。新代码从提交 `87dc130` 起改用去 Cookie 的 `stream_headers`，随后仅在匿名握手失败时回退到登录态。
- 这正是当前回退策略的盲点：匿名握手成功不等于弹幕流完整，因此永远不会触发 Cookie 回退。若以录播弹幕完整性为优先，可靠方案应改为“弹幕 WebSocket 登录态优先、Cookie 不可用时匿名兜底”；房间状态轮询、普通 HTTP 仍可保持匿名/轮换，避免扩大带账号请求面。
